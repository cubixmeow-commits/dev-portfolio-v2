## Flagship products

current product line

Two original products showing the full build — product concept, interaction design, application architecture, data model, safety boundaries, deployment path, and ongoing iteration.

flagship / 01

SousMeow

MERGED · v1 complete

Thesis: A platform for turning complex AI tasks into guided, reusable workflows that work with the AI subscriptions people already have.

Instead of a giant prompt, SousMeow walks a person through the task one dependable step at a time: copy a prepared prompt, run it in ChatGPT, Claude, or Gemini, paste the response back, confirm human checks, progress, and finally export a complete Project Kit.

  • Every pasted response is versioned; approvals bind to the exact content. Editing or restoring creates a new version and resets checks.
  • Deterministic progression: Pantry → Recipe steps → checks → export. SQLite or MySQL behind one config switch. CSRF, ownership enforcement, PDO prepared statements.
  • The core product never calls an AI API; the human carries responses across the trust boundary on purpose.

flagship / 02

Rally

OPEN · building

Thesis: Turn passive health tracking into structured competition.

Rally runs timed head‑to‑head matchups using normalized daily metrics and explicit rules for category scoring and winners. The work focuses on the scoring model, matchup state, and data provenance before expanding the interface.

  • Two competition types (Baseline and Classic) with daily snapshots, tie handling, and published rules.
  • Per‑source series and baselines; no silent merging of Apple Watch and phone steps.
  • Match state, category results, and shareable summaries designed to avoid medical claims.

## Operational systems

supporting proof

Supporting systems, shown working. Every measurable claim is inspectable in the live app or case study.

project 02

it-asset-inventory-tracker

MERGED · live

A deterministic system of record for hardware and software assets, built for environments where the audit trail is the product. Status changes flow through a PHP state machine that makes invalid transitions impossible to record — not merely hidden behind a disabled button — and every mutation is logged with who, what, when, and the previous value.

  • Lifecycle enforced by a state machine: an asset cannot jump from RETIRED to ASSIGNED because no code path exists to write it.
  • Append-only audit log: every change records actor, field, old value, new value, timestamp.
  • Backups are restore-tested, not assumed: nightly dumps plus timed restore drills.
  • An AI advisory layer reads aggregates and writes recommendations to its own tables. It has no write path to the inventory.

project 03

service-request-tracker

OPEN · building

A trouble-ticket pipeline for IT service requests: intake, triage, assignment, resolution. Priorities and SLA timers are set by rules a human can read; escalation is a logged state change, not a Slack message someone hopes gets seen.

  • The AI triage step is simulated in the demo and disclosed as simulated in the UI; the seam is designed now so a real model can slot in later without changing the trust boundary.
  • AI suggests a priority and category; a human confirms. The suggestion is stored beside the human decision, so the model's accuracy is auditable after the fact.
  • Every ticket carries its full state history: who moved it, when, and why.

project 04

helpdesk-ops-toolkit

OPEN · active

A department-agnostic IT support system built to demonstrate government support competencies end to end: public ticket intake, an agent queue with triage and assignment, reporting, an asset tracker, and a plain-language knowledge base — on the same stack that actually runs things.

  • PHP 8 + MySQL web app: a public submit-a-ticket view and an auth-gated agent queue with filters, assignment, and logged resolution notes. PDO throughout, prepared statements, CSRF-guarded forms.
  • A Java (JDBC) CLI companion generates the weekly ops report from the same database — the same query logic delivered in two languages, the way real IT teams pair a web system with a support script.
  • Reporting is hand-rendered inline SVG, the asset tracker exports CSV, and the knowledge base renders Markdown — no framework, no external requests, its own civic design.

## AI integration policy

the governor

Systems of record stay deterministic. The AI layer reads aggregates, writes recommendations to its own tables, and never touches the data. Framed the way you'd read a diff — what's added versus what's refused.

+ what the AI layer does
  • +Runs on a schedule, from cron
  • +Reads pre-aggregated metrics, never the raw schema
  • +Writes recommendations to its own reports table
  • +Flags anomalies for a human to review
− what it never does
  • −Assign or modify assets
  • −Delete or write records
  • −Override business rules
  • −Act without human review

This pattern runs live in the Inventory Tracker. v1 proved the response contract by hand: export a brief, paste it into any model, import the JSON reply. v2 automated the same contract without moving the trust boundary. If the API is down, inventory operations continue; if leadership asks what AI changed, the answer is always nothing, by design. Read the case study ↗
